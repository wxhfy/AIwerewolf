# 1. 项目最终目标

本项目的核心不是做一个普通狼人杀游戏，而是做一个：

> **可自主完成一局狼人杀的 Agent Team，并进一步具备可评测、可复盘、可量化、可轻量自进化能力的 AI 狼人杀多智能体系统。**

为什么选择狼人杀：

- 狼人杀具备多角色；
- 存在强信息不对称；
- 同时需要阵营协作与阵营对抗；
- 既有规则推理，也有社交博弈；
- 非常适合检验 Agent Team 的综合能力。

请判断当前仓库是否支撑以下目标：

1. **角色 Agent**  
   支持狼人、预言家、女巫、猎人、村民等常见角色，每个角色具备对应行动、策略和身份差异。

2. **对局引擎**  
   支持回合流转、夜间行动、白天发言、投票、死亡结算和胜负判定。

3. **信息隔离**  
   每个 Agent 只能看到自己身份允许的信息，不能偷看其他玩家真实身份、狼队信息、预言家查验结果或赛后真相。

4. **可观测性**  
   对局日志可查看，关键决策可追溯，能够复盘每个 Agent 在每回合的输入、输出和决策原因。

5. **评测与复盘**  
   不只看胜负，还能评价发言、投票、技能、阵营协作、关键错误和反事实结果。

6. **轻量自进化**  
   复盘结果不能直接当真理，必须经过可信度、权限和适用条件过滤后，才能进入策略知识库并被下一局 Agent 检索使用。

7. **加分能力**  
   提供前端 UI，支持观战或人机混战；支持 Leaderboard、批量对局、A/B 验证、版本回溯。

请用一句话概括项目主线。建议判断方向：

> 我们做了一个 AI 狼人杀多智能体系统，不只会玩，还能复盘每个关键决策，通过反事实评估定位 bad case，并把经过可信度、权限和适用条件过滤的策略知识回流到下一代 Agent。

请结合仓库实际情况判断这句话是否成立。如果不完全成立，请指出缺口。

# 2. 工作模式：审查 + 小改 + 大改建档

本任务不只是评分验收。你需要将发现的问题分成四类处理。

## 2.1 改动分级

| 等级 | 类型 | 允许动作 | 示例 |
|---|---|---|---|
| S0 | 只记录，不修改 | 写入报告即可 | 代码风格建议、长期优化想法 |
| S1 | 安全小改 | 可以直接修改并提交 | 文档补充、注释修正、配置默认值修正、测试说明补全 |
| S2 | 小范围代码修复 | 可以直接修改，但必须最小 patch、可测试、可回滚 | 明显 typo、错误 fallback 顺序、日志字段缺失、测试断言补充 |
| S3 | 架构级大改 | 不直接修改，必须写方案和任务清单 | 重构 Skill 协议、重写引擎、改 DB schema、大规模迁移、前端重做 |

## 2.2 可直接改的内容

如果发现以下问题，且影响范围明确，可以直接小步修复：

- 文档与实际代码路径不一致；
- README 或 docs 中的明显错误表述；
- fallback 方向写反；
- 测试命名不清晰；
- 缺少明显的验收说明；
- 配置示例缺失但能安全补充；
- 日志字段文档缺失；
- 已有测试可以轻量补充断言；
- 注释误导；
- 评测报告模板缺字段；
- 大改方案文档缺失。

## 2.3 不要直接改的内容

以下内容不要贸然直接改代码，除非任务明确要求并且已有完整测试：

- 重构游戏引擎状态机；
- 重写角色技能系统；
- 迁移数据库模型；
- 改变现有规则结算顺序；
- 更换 Agent 架构；
- 重写 Retrieval / Knowledge Store；
- 大幅修改 Prompt 格式；
- 重做前端；
- 引入新的外部依赖；
- 影响 API 兼容性的修改。

这些必须写入：

```text
docs/system_improvement_backlog.md
```

或报告中的“大改待办”章节。

# 6. 解释设计方案为什么这么设计

不要只罗列模块。请解释设计背后的原因。

至少说明以下设计：

## 6.1 为什么需要 PlayerView / Visibility 信息隔离？

请解释狼人杀的信息不对称为什么是核心挑战，以及为什么不能直接把完整 GameState 传给 Agent。

必须回答：

- 如何防止狼人看到预言家查验？
- 如何防止村民看到狼队名单？
- 如何防止 Memory / Retrieval / Prompt 泄露 hidden truth？
- 为什么需要 final_agent_input 级别测试？
- 如果当前项目只测 PlayerView，不测 final_agent_input，请将 final_agent_input 测试列为 P0 或 P1 改进项。

## 6.2 为什么需要 BeliefTracker？

请解释 Agent 不能知道真相，只能维护“自己相信的身份概率”。

必须区分：

```text
真实身份 Truth
Agent 信念 Belief
公开声称 Claim
历史行为 Evidence
```

## 6.3 为什么需要 RoleStrategyCard？

请解释人格风格和策略决策的区别。

```text
Personality Layer：控制怎么说
Strategy Layer：控制做什么
```

说明为什么不同角色不能只共用一个 Prompt。

## 6.4 为什么需要 DecisionTrace？

请解释为什么不能只保存原始 LLM 输出，而要记录：

- visible_facts；
- candidate_actions；
- chosen_action；
- confidence；
- rationale；
- retrieved_strategy_ids；
- prompt_hash；
- model / token / latency / cost。

如果当前仓库没有结构化 DecisionTrace，请把它列为 P1 改进；如果已有日志但不完整，列出可小改字段。

## 6.5 为什么需要 CounterfactualAnalyzer？

请解释为什么“只看胜负”不足以评价 Agent 决策。

示例：

- 好人赢了，但女巫可能仍然做错过；
- 狼人输了，但某只狼的倒钩策略可能是正确的；
- 某个投票动作可能是关键转折点。

## 6.6 为什么需要 L0-L4 知识可信度分层？

请解释为什么复盘不一定正确，错误复盘直接入库会污染下一局 Agent。

必须说明：

```text
不是所有复盘都可信；
不是所有可信知识都可见；
不是所有可见知识都适用于当前局面；
不是所有知识都应该永久保留。
```

如果当前仓库中有 StrategyKnowledgeDoc 但没有 confidence / access_control / applicability，请列为 P0 或 P1 风险。

## 6.7 为什么主线应该是“评测 + 复盘”？

请判断当前项目更适合主打哪个进阶方向：

- A. 通用 Agent；
- B. 评测 + 复盘；
- C. 自进化 Agent。

请说明理由，并指出如果主打错误方向会有什么风险。

# 8. 进阶课题要求与验收方式

进阶课题三选一。请检查当前项目最适合选择哪个方向，并说明原因。

## 8.1 A. 通用 Agent

### 核心问题

Agent 能否根据目标自主生成 / 修改代码、测试、部署？

### 具体要求

- 能读懂自身代码；
- 能改代码并提交；
- 能自动构建测试重启；
- 失败可回滚；
- 过程可审核。

### 验证方式

以“玩一局狼人杀”为目标，系统自主演化出可运行的 Agent Team。

### 请判断

当前项目是否真的具备：

```text
读代码 → 改代码 → 自动构建测试 → 失败回滚 → 审计记录
```

如果没有，请不要把它作为主赛道。

## 8.2 B. 评测 + 复盘

### 核心问题

如何系统化评价 Agent 决策质量，而不只看胜负？

### 具体要求

- 多维评测：发言 / 投票 / 技能；
- 关键决策复盘；
- 反事实推演；
- 结构化报告；
- Leaderboard。

### 验证方式

构造含明显失误的对局，系统能定位失误并给出改进建议。

### 请检查

- 是否能生成 ReviewReport；
- 是否有 PerStepScorer；
- 是否有 CounterfactualAnalyzer；
- 是否能定位 bad case；
- 是否有 Leaderboard；
- 是否能区分不同模型 / Agent 版本能力；
- 是否能把明显缺失的报告模板直接补齐。

## 8.3 C. 自进化 Agent

### 核心问题

Agent 能否在多局迭代中持续提升胜率？

### 具体要求

- 自动“对局 → 分析 → 调整 → 再对局”闭环；
- 策略适应性；
- A/B 对战验证；
- 版本回溯。

### 验证方式

终局 Agent 与初始 Agent 对战 20 局，胜率显著提升。

### 请检查

- 是否有 StrategyKnowledgeDoc；
- 是否有知识回流；
- 是否有知识可信度分层；
- 是否有 access control；
- 是否有 applicability；
- 是否有 A/B Tournament；
- 是否有版本记录；
- 是否真的有 20 局对战提升数据。
