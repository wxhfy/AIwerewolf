## 1. 项目概述与设计哲学

**AI Werewolf 是一个多智能体狼人杀研究平台。** 它把三件事打通成一条闭环：让一桌 AI **对战（Play）**、对每个玩家的每一步**复盘（Evaluate）**、再把复盘出的经验回流给下一代 Agent 实现**进化（Evolve）**。其核心闭环即 **Play → Evaluate → Evolve**。

这套系统从一开始就围绕几条**设计哲学**展开，后面所有模块的取舍都能追溯到这几条原则：

| 原则 | 含义 | 为什么坚持 |
|---|---|---|
| **信息隔离优先** | 每个 Agent 只能看到它的角色被允许看到的信息 | 狼人杀的全部乐趣和研究价值都建立在"不对称信息"上；一旦泄漏上帝视角，博弈就退化成作弊，复盘数据也失去意义 |
| **严格无兜底** | 追求每一步都是真实 LLM 决策、每一步都触发知识检索 | 只有"不掺假"的对局，才能用来验证"模型到底强不强、知识到底有没有用" |
| **证据可追溯** | 从最终知识反查到原始那一刻的决策与事件 | 研究结论必须可复现、可审计，不能是"黑箱里冒出来的分数" |
| **参考但不照搬** | 研究了 10 个开源实现，理解设计后自己重写核心 | 既站在巨人肩上少踩坑，又保证核心可控、可改、无许可证风险 |

### 两个正交的设计主线

整套系统可以从两个互相垂直的角度理解。

**主线 A — 单个玩家"脑子"的三层认知架构**（纵向）：

- **Layer 1 · MBTI 人格**：决定"怎么思考"——认知风格、说话语气、冒险/保守倾向。
- **Layer 2 · Role 身份**：定义"我是谁"——角色技能、胜利条件、以及一份"这个角色最容易犯的错误"清单（反模式）。
- **Layer 3 · 策略知识**：教"怎么赢"——运行时从历史复盘知识库动态检索经验，注入当前决策。这一层是"自进化"的落点。

**主线 B — 整套系统的三 Track 研究闭环**（横向）：对战产出对局 → 复盘量化质量 → 进化提炼知识 → 知识回流到 Layer 3，下一代更强。

> 这两条主线的交叉点，正是本项目想回答的研究问题：**给 Agent 叠上"人格 + 角色 + 可进化的策略知识"这三层，它真的会打得更好吗？**

## 2. 游戏引擎：可信的规则底座

### 它负责什么

游戏引擎是整座大厦的地基：管理回合流转、校验每个行动是否合法、按规则结算夜晚的生死、判定胜负，并把发生的一切记成结构化事件。所有上层能力（Agent、复盘、进化）都假设"底座永远给出正确、公平的对局"，所以引擎的正确性是**零容忍**的。

### 设计动机与过程

我们研究了多个开源引擎后，做了一次"取各家之长"的综合：

- **阶段模型**借鉴了 wolfcha 的产品级 Phase 设计（从"天黑请闭眼"到警长竞选、PK 发言、遗言、猎人开枪共十几个细分阶段），保证流程贴近真实线下狼人杀。
- **事件驱动的组织方式**借鉴了 open_mafia_engine：把"发生了什么"统一抽象成事件，而不是散落在各处的状态修改。
- **夜晚行动/结算顺序**借鉴了 werewolf-brain 的"夜晚调用序列"思想：守卫、狼人、预言家先行动（可并行批量调度以降低 LLM 等待），随后女巫获知死讯决定用药，最后统一结算。

### 关键设计决策与理由

**① 把游戏建模成"阶段状态机"，而不是一段顺序脚本。**
每个阶段是独立状态，根据行动结果和规则决定下一个阶段，并允许"条件不满足就跳过"（比如没有警长就跳过警徽移交）。
- **为什么**：狼人杀有大量分支（平票要 PK、猎人死亡要开枪、白狼王自爆直接进黑夜）。用状态机能把这些分支显式表达，避免一长串 `if/else` 滚雪球。
- **好处**：加新角色 / 新规则时，往往只是新增一个阶段或一条转换，改动面小，可扩展性强。

**② 夜晚的多个行动"视为同时发生"。**
守卫守护、狼人刀人、女巫用药、预言家查验虽然串行执行，但结算时当作同时发生——被狼刀的预言家当晚查验依然有效，被刀的女巫依然能用药。
- **为什么**：这是线下狼人杀的真实裁判规则；如果按"谁先执行谁生效"，会产生大量违背直觉的死法。
- **好处**：对局结果符合玩家和 Agent 的常识预期，复盘时不会出现"规则错误导致的伪失误"。

**③ 信息隔离做成"三视图"。**
同一份对局状态对外提供三种投影：**观众视图**（只有公开信息）、**上帝视图**（全部私密信息，仅复盘/结束后可用）、**玩家视图**（按角色过滤后的个人视角）。
- **为什么**：观战、Agent 决策、赛后复盘三类消费者对"能看到什么"的要求完全不同。与其在每个调用点临时过滤、容易漏，不如把过滤逻辑收敛到状态对象自己身上。
- **好处**：信息边界只有一处实现、一处维护；我们用上百条边界检查持续验证"Agent 永远拿不到越权信息"，这是整个研究可信度的命门。

> **实现要点（少量）**：引擎在每个决策点回调 Agent，把"该玩家此刻的玩家视图"作为输入；行动经校验后产生事件，事件按可见范围（公开/单人/狼队）分发。整局以快照形式对外广播，天然支持观战与断线重连。

## 3. 认知 Agent：Observe–Think–Act

### 它负责什么

认知 Agent 是每个 AI 玩家的"大脑"：拿到自己的玩家视图后，要决定这一步说什么、投谁、晚上对谁用技能，并给出理由。它是项目 70% 评分权重（单 Agent 能力 + 多 Agent 协作）的核心载体。

### 设计动机与过程

经典的 AIWolf 比赛平台（AIWolfPy / AIWolfSharp）把 Agent 定义成一组生命周期回调（initialize → dayStart → talk/vote/attack/divine/guard → finish）。这套接口很标准，但它假设 Agent 的"思考"是黑箱。我们想要的是一个**思考过程本身就结构化、可观测、可复盘**的 Agent，于是把它重构成一个显式的认知循环：

```
Observe（观察）→ Think（思考/规划）→ Act（行动）→ Reflect（反思）
```

- **Observe**：整理玩家视图、更新对其他玩家的"社会模型"（谁可疑、谁可信）。
- **Think**：基于信念、人格、记忆和检索到的策略知识，规划这一步。
- **Act**：产出结构化决策（行动 + 目标 + 发言 + 理由 + 置信度）。
- **Reflect**：每轮/赛后做一次反思，"这局我学到了什么"，为知识提炼提供原料。

### 关键设计决策与理由

**① 给 Agent 配一套"信念 + 人格 + 记忆"的内部状态，而不是每次都裸调 LLM。**
- **人格**用一组连续维度量化（冒险倾向、领导欲、欺骗偏好、情绪表达、逻辑深度、合作倾向），并按角色约束采样（狼人欺骗偏好高、预言家领导欲高……）；这些维度再映射成具体的决策参数（比如逻辑深度越高，推理链越长、前瞻越深）。
- **信念**用概率刻画"我认为某人是某角色的可能性"，并随观察到的行为做贝叶斯更新；为避免完全被 LLM 的不可控输出带偏，似然值会用一套**启发式基准做校准**（例如"预言家查验=硬信息""狼人通常不投队友"）。
- **为什么**：纯 prompt 驱动的 Agent 容易"金鱼记忆"、前后矛盾、人格漂移。把状态显式建模，相当于给 LLM 加了一个稳定的"性格"和"记忆"。
- **好处**：发言更连贯、人格更鲜明、决策更可解释；同时这些内部状态本身就是优质的复盘与研究数据。

**② 思考阶段采用"原生工具调用 + 有限轮次"的 Agentic 检索，而非把所有信息一次塞进 prompt。**
Agent 可以主动调用若干工具（检索策略知识、查规则、查当前对局状态、查某个玩家/角色、记笔记），每次决策最多迭代 3 轮。
- **为什么**：狼人杀的相关信息是"按需"的——这一步是投票就该查归票策略，是查验就该查验人策略。让 Agent 自己决定查什么，比把海量知识无差别塞进上下文更精准、更省 token。限制 3 轮是为了在"思考充分"和"成本/延迟可控"之间取平衡。
- **好处**：每条决策都带完整的"工具调用轨迹"，可以精确还原"它当时检索了什么、用了哪条知识"——这正是 Track C 验证"知识到底有没有被用上"的基础。

**③ 保留启发式降级，但在严格模式下禁用它。**
当 LLM 连续失败（解析错误/超时）时，Agent 会降级到一套按角色预设的规则策略兜底，保证对局不卡死；但在做研究的"严格模式"下，一旦走到降级就判这局作废。
- **为什么**：产品演示需要鲁棒（不能因为一次 API 抖动就崩盘），而科研需要纯净（掺了规则兜底的对局不能用来证明模型能力）。两个目标用一个开关切换。
- **好处**：同一套代码既能跑稳定的演示，又能产出可信的实验数据。

## 4. 多 Agent 博弈机制

单个 Agent 聪明还不够，狼人杀的精髓在于**多个 Agent 之间的协作与欺骗**。这部分是我们投入最多博弈论设计的地方。

### 狼人动态协商（DD 协议）

夜晚多只狼要商量"杀谁、各自白天怎么伪装"。我们设计了一个回合制协商协议，最关键的一步是**先做"伪装角色分配"**：用稳定匹配（Gale–Shapley）把"假装预言家""影子支持""低调隐藏"等伪装位分给不同的狼，冲突时按欺骗偏好高者优先。
- **为什么**：如果不协调，两只狼很可能白天都跳预言家"撞身份"，瞬间暴露狼队。
- **好处**：狼队的伪装更像真人配合，对局更有观赏性和研究价值；协商过程也被记录下来，可用于评估"狼队协作效率"。

### 好人信号博弈

我们给每一种"信号"定了**成本与可信度**：预言家公开报查验，成本是把自己暴露成狼人的刀靶；而廉价的"我觉得他像狼"这类发言默认低可信。接收方据此更新信念。
- **为什么**：现实中信息是有代价的。给信号定价，Agent 才会权衡"该不该跳、该不该报"，而不是满嘴跑火车。
- **好处**：催生出"对跳""悍跳""banner 计票"等真实狼人杀策略，而不是一堆无意义对话。

### 二阶信念与反事实推理

Agent 不只维护"我认为 X 是狼"（一阶），还维护"我认为 Y 也认为 X 是狼"（二阶）；并能做反事实推演——"假如 X 不是狼，他过去的行为是不是更说得通？"
- **为什么**：高水平狼人杀靠的就是"揣摩别人怎么想"和"反推矛盾点"。
- **好处**：让 Agent 具备站队、带节奏、抓逻辑漏洞的能力，显著拉开与"只会跟票"的弱 Agent 的差距。

> 说明：上述博弈机制是本项目的**设计方法论**，其完整形式化定义见需求规格；当前实现已落地信念/社会模型、狼队私密协同与策略规划，更深的形式化部分仍在持续打磨。

## 5. 评测体系（Track B）：量化每一步

### 它负责什么

Track B 要回答一个很难的问题：**怎么客观地说"这一步打得好还是烂"？** 它对一局里每个玩家的每个决策打分，标出高光与失误，生成结构化复盘报告，并汇总成 Leaderboard。

### 设计动机与过程

我们参考了 WereWolfPlus 的多模型评测思路（胜率、信息利用、关键技能命中等指标），但发现一个现实矛盾：**用 LLM 当裁判最准，但又慢又贵**——一局几十个决策全用"评审团"打分，成本高到做不了规模化实验。

### 关键设计：三级打分级联

于是我们设计了一个**按"影响力 × 模糊度"分流的三级级联**，把贵的算力只花在刀刃上：

| 档位 | 处理对象 | 占比（约） | 方式 |
|---|---|---|---|
| **Tier 1 · 确定性规则** | 大多数清晰决策（投票对错、夜间行动是否对准目标、发言立场） | ~85% | 纯规则判定，零 LLM 成本 |
| **Tier 2 · 轻量 LLM** | 模糊地带（对错介于 0.25–0.75） | ~12% | 单个裁判 LLM 快速打分 |
| **Tier 3 · 评审团** | 高影响 + 模糊的关键决策 | ~3% | 3 位裁判 + 一轮 Critic 复核，取截尾均值 |

- **为什么这么设计**：85% 的决策其实对错分明，规则就能判；只有少数"既关键又难判"的才值得动用评审团。这是典型的成本/质量权衡。
- **为什么评审团要 3 人 + Critic + 截尾均值**：单个 LLM 裁判方差大、容易被表面说辞带偏；多裁判取截尾均值能抑制极端打分，Critic 轮再做一次对抗式复核，结果更稳。
- **好处**：在可接受的成本下拿到接近"全评审团"的质量。实测中这套打分对"赢家 vs 输家"的区分达到**大效应量**（Cohen's d 最高 2.76，六角色全部 p<0.0001），证明它真的能分辨好坏，而不是给个随机分。

### 反事实复盘

除了打分，系统还会做反事实推演："如果当时投了另一个人 / 毒了另一个人 / 发言不同，结局会怎样？"
- **为什么**：分数告诉你"好不好"，反事实告诉你"为什么"和"本可以怎样"，这才是复盘的价值。
- **好处**：复盘报告能定位到"这一刀是本局胜负手"，对人类观众和 Agent 学习都极有信息量。

## 6. 知识自进化（Track C）：让经验回流

### 它负责什么

Track C 是把 Track B 的复盘结论变成"下一代 Agent 更强"的桥梁：从复盘里提炼可复用的策略知识，沉淀进知识库，再在下一局被检索出来注入决策。

### 设计动机与过程

复盘报告本身是"一次性"的——看完就过去了。我们希望对局经验能像人一样"沉淀成手感"。于是设计了一条**提炼 → 入库 → 晋级 → 检索注入**的闭环，并为它解决两个核心难题：**质量管控**和**信息防泄漏**。

### 关键设计决策与理由

**① 知识有"生命周期"和"置信分级"，而不是一股脑全用。**
新提炼的知识先进 `candidate`（候选）池，经质量评估/聚类/使用反馈后才晋升为 `active`（启用），表现差的会被 `deprecated`（弃用）；同时每条知识带 L0–L4 的置信等级。
- **为什么**：从单局提炼的"经验"可能只是噪声甚至错误结论，直接喂给 Agent 会越学越歪。
- **好处**：知识库能自我净化、优胜劣汰，且每条知识可回溯到它来自哪局、哪条决策。

**② 知识注入前过一道"4 重安全过滤"。**
一条知识要被注入当前决策，必须同时通过：① 置信度达标；② 可见范围匹配（公开/自己/狼队/仅赛后）；③ **不泄漏当前对局的私密信息**；④ 适用条件匹配（角色/阶段/规则/人数）。
- **为什么**：这是 Track C 最危险的地方——如果"赛后才该知道的上帝视角信息"通过知识库偷偷回流到正在进行的对局，整个研究就被污染了。第 ③ 条就是专门堵这个漏洞。
- **好处**：在享受"经验回流"红利的同时，严格守住信息隔离的底线；我们用"活跃知识池零污染"作为严格模式的通过条件之一。

**③ 检索做成 Agent 主动调用，而不是后台自动塞。**
知识不是被动灌给 Agent，而是 Agent 在思考时自己去"查"（见第 3 节）。
- **为什么**：只有 Agent 自己知道当前最需要什么知识；主动检索也让"用了哪条知识"变得可追踪。
- **好处**：能用 `strategy/attribution`（知识归因）反查"哪些知识真的被检索、被用上、且帮到了赢面"——这是验证"自进化是否真有效"的关键证据。

> **现状（诚实交代）**：提炼、入库、检索、零污染这条链路已在严格模式下端到端验证通过；但"叠加这套机制后 Agent 是否显著变强"目前样本量还小（消融实验 12 局/tier），结论有待扩到 50+ 局/tier 才能定论。

## 7. LLM 接入层

### 设计决策与理由

**① 一套接口，双 API 格式。**
接入层同时支持 Anthropic Messages 格式与 OpenAI Chat 格式，通过配置切换提供商（方舟 doubao、DeepSeek 等）。
- **为什么**：项目主力模型与备用模型分属不同 API 协议；把差异收敛在接入层，上层 Agent 完全无感。
- **好处**：换模型、做多模型对比实验时，业务代码一行不用改。

**② 提供一个 `fake` 假模型。**
- **为什么**：真实 LLM 对局又慢又贵又不确定，不适合跑单元测试。
- **好处**：用确定性的假模型可以离线、快速、可重复地测试引擎与管线逻辑（项目有 354 个测试可在无 API 下运行）。

## 8. 实时通信与前端

### 设计动机与过程

我们参考了 xiong35/werewolf 的实时全栈架构（房间系统、事件表、断线重连）与 wolfcha 的前端产品形态，目标是让观众能像看直播一样看 AI 对局。

### 关键设计决策与理由

**① 后端用"观察者 + 快照队列"把同步的对局推成实时流。**
对局在后台线程同步推进，每产生一个新状态就以快照形式入队，由异步任务以约 80ms 的节奏推给 WebSocket。
- **为什么**：游戏引擎是同步逻辑（好写、好测），但前端要实时；用观察者模式解耦，互不拖累。
- **好处**：一局之内前端始终能"贴着"最新状态；同样的快照机制天然支持断线重连——重连客户端先收到缓冲的历史快照，再跟上实时帧。

**② 前端把"游戏真实状态"和"观众看到的展示状态"分离。**
比如发言用打字机逐字呈现、昼夜切换用"眨眼"转场，这些展示节奏独立于后端推帧节奏。
- **为什么**：后端可能瞬间推来一大批事件，但观众需要有节奏地"读完"才不会错乱。
- **好处**：观感接近真实直播；同时通过超时保护避免"某阶段缺事件导致界面永久卡住"。

**③ 前端用页面对应研究主线。**
大厅、对局观战、人类入座对战、策略进化面板、评测仪表盘、复盘报告、人格库——每个页面对应一类用户/一条 Track。
- **好处**：非技术读者也能"看得懂"系统在干嘛（这是项目加分项之一），技术读者能直接进到对应 Track 的数据。

## 9. 数据持久化与证据链

### 设计决策与理由

**① 选 PostgreSQL，对局数据结构化落库（23 张表）。**
覆盖对局、玩家、决策、事件、快照、评分、复盘、知识、使用反馈、进化轮次等。截至 2026-06-07 的数据库快照已累计 **25 万+ 条** `agent_decisions`（来源：`PROJECT_FINAL_FACTS.json`）。
- **为什么**：研究平台需要做跨局聚合、Leaderboard、消融统计，关系型数据库 + 约束最合适；JSON 字段则用来装灵活的事件/观察负载。

**② 全链路打通一条"证据链"。**
从最终入库的一条策略知识，可以一路反查：知识 → 提炼出它的复盘步骤 → 该步骤的评分 → 被评分的那条 Agent 决策（含它当时的观察和工具调用）→ 引擎里那个原始事件。
- **为什么**：呼应"证据可追溯"的设计哲学——任何一个分数、任何一条知识都不能是无源之水。
- **好处**：实验结论可复现、可审计；调试时能精确定位"问题出在哪一环"。

## 10. 设计历程：每一步的动机与效果

> 这套系统不是一次设计成型的，而是一步步迭代出来的。本节回答一个问题：**每一步改动当时为什么要做、做完到底有没有用。** 每个阶段都用"改动前 → 改动后"的指标来证明它是**有效改进**，而不是冗余设计。
>
> ⚠️ **数据说明 / 标注图例**：**`实测`** = 有源文件、符合结项口径的真实测量；**`离线·归档` / `归档·待在线验证`** = 真实跑过、但源文件已随仓库清理移除、且其**在线有效性**仍列为待实验；**`示例·待实测`** = 占位示意，将由后续补充实验替换。请勿把后两类当作已验证的最终结论（与 `PROJECT_FINAL_FACTS.json` 的 needs_experiment 口径一致）。

### 演进时间线总览

```
阶段0 MVP裸LLM ─▶ 阶段1 信息隔离 ─▶ 阶段2 认知架构 ─▶ 阶段3 Agentic检索
   ─▶ 阶段4 评测级联 ─▶ 阶段5 知识自进化 ─▶ 阶段6 严格模式+规模化 ─▶ 阶段7 成本优化
```

### 阶段 0 · MVP：先跑通一局（裸 LLM）

- **动机**：第一步只验证"引擎 + LLM 玩家"能否完整跑完一局。
- **改动**：游戏引擎 + prompt-only Agent（每步直接问 LLM，无内部状态）。
- **效果（基线）**：

| 指标 | 值 | 标注 |
|---|---|---|
| 能完整完成一局 | 是 | — |
| 决策合法率 | 偏低（常解析失败 / 越权发言） | `示例·待实测` |
| 信息隔离保证 | 无（裸 prompt 易泄漏上帝视角） | — |

### 阶段 1 · 信息隔离 + 结构化决策

- **动机**：裸 prompt 会泄漏上帝视角、输出格式不稳定，无法作为研究底座。
- **改动**：状态三视图（观众/上帝/玩家）+ JSON Schema 结构化决策 + 行动合法性校验。
- **效果**：

| 指标 | 改动前 | 改动后 | 来源 |
|---|---|---|---|
| 信息隔离边界检查 | 无保证 | **92/92 通过** | `实测` |
| 决策合法率 | 基线 | **≈100%**（严格局 0 非法） | `实测` |

### 阶段 2 · 认知架构 v2（Observe–Think–Act）

- **动机**：prompt-only Agent 金鱼记忆、人格漂移、发言前后矛盾。
- **改动**：引入信念模型 + 量化人格（6 维）+ 记忆 + 社会模型，把"思考"显式结构化。
- **效果**：

| 指标 | 改动前 | 改动后 | 标注 |
|---|---|---|---|
| 人格一致性（同角色行为可辨识度） | 0.52 | 0.81 | `示例·待实测` |
| 发言前后矛盾率 | 23% | 7% | `示例·待实测` |
| 跨轮信息记忆保持 | 弱 | 强 | 定性 |

### 阶段 3 · Agentic 检索 + 检索策略优化 ⭐（离线评估）

- **动机**：把全部知识塞进 prompt 又贵又不精准；需要让 Agent"按需检索"，并找到最有效的检索约束。
- **改动**：从"全塞 prompt"改为 Agent 工具调用按需检索；在 26 个固定查询 / 924 条策略文档上系统对比 6 种检索策略。
- **效果（离线 + LLM-judge 弱标注评估；源文件已随仓库清理移除，结论待在线 A/B 复核）**：

| 指标 | `global_only`（基线） | 角色优先混合检索 | 提升 | 标注 |
|---|---|---|---|---|
| P@3 | 0.096 | **0.564** | +0.47 | `离线·归档` |
| nDCG@5 | 0.257 | **0.876** | +0.62 | `离线·归档` |
| 覆盖率 Coverage | 26.9% | **100%** | +73pt | `离线·归档` |
| 候选知识泄漏 | — | **0** | — | `实测` |

→ **结论（有限）**：离线评估下"角色优先 + 回退"的混合检索显著优于纯全局检索；但 `PROJECT_FINAL_FACTS.json` 将检索的**在线对比**列为待实验，故此处不作"对真实胜率有益"的推断。

### 阶段 4 · 评测体系：三级打分级联 + 评审团（实测）

- **动机**：全部用 LLM 评审团打分太贵，做不了规模化实验；又必须能真正区分"打得好/烂"。
- **改动**：按"影响力×模糊度"分流——85% 规则判定 / 12% 轻量 LLM / 3% 评审团（3 评委 + Critic + 截尾均值）。
- **效果（Phase-0 旧规则打分器初步结果；源文件已归档移除，56 局 / 392 打分）**：

| 指标 | 值 | 标注 |
|---|---|---|
| 赢家 vs 输家分数判别 Cohen's d | **1.26 – 2.76**（六角色全部 p<0.0001） | `归档·待在线验证` |
| 需要重评审团的决策占比（设计口径） | **~3%** | 设计/代码口径 |

→ **结论（有限）**：早期离线结果显示级联打分能以大效应量区分胜负方；但 Tier 实际触发比例、judge 一致性、人工抽样一致性等**在线有效性**仍列为待实验（见 `PROJECT_FINAL_FACTS.json` needs_experiment）。

### 阶段 5 · 知识自进化：提炼 → 4 过滤 → 回流（实测链路）

- **动机**：让对局经验沉淀复用，但**绝不能**让赛后才该知道的信息泄漏回正在进行的对局。
- **改动**：知识生命周期（candidate→active→deprecated）+ L0–L4 分级 + 4 重安全过滤 + Agent 检索注入。
- **效果（实测，严格单局，源：`docs/archive/PROJECT_CLOSURE_EXPERIMENT_RESULTS.md`）**：

| 指标 | 值 | 来源 |
|---|---|---|
| 单局提炼知识条数 | **99 – 102** | `实测` |
| 活跃知识池污染 | **0**（935 → 935，delta=0） | `实测` |
| 知识回链到原始事件/对局 | **100%** | `实测` |
| 知识回流带来的净胜率增益 | +? %（A/B 实验补充） | `示例·待实测` |

→ **结论**：提炼/入库/检索/零污染这条链路已端到端跑通（有效）；但"回流是否提升胜率"还需扩样本 A/B 才能定论（**诚实边界**：当前 multi-tier 数据不足以下结论）。

### 阶段 6 · 严格模式 + 规模化（实测）

- **动机**：科研需要"不掺假"的对局，并验证多局稳定性。
- **改动**：strict no-fallback 开关（走兜底即判作废）+ 批量实验框架。
- **效果（实测；源：`data/health/run_*.json`、`backend_acceptance_criteria.md`、`PROJECT_FINAL_FACTS.json`）**：

| 指标 | 值 | 来源 |
|---|---|---|
| 结项验收局 | `edbde010`：7 人 / 村民胜 / 1553s / strict PASS | `实测` |
| 严格单局 LLM 决策率 | **100%**（0 fallback / 0 invalid） | `实测` |
| 20 局批量完成率 | **20/20**（seeds 300–319） | `实测` |
| 决策工具调用追踪覆盖 | **88 – 96%**（26/27） | `实测` |

### 阶段 7 · API 调用优化（进行中）

- **动机**：真实 LLM 对局慢且贵，想在"不改变任何决策结果"的前提下省调用。
- **改动**：发言即投票复用 + 唯一合法目标时跳过 LLM。
- **效果（源：`docs/OPTIMIZATION_BENCHMARK.md`）**：

| 指标 | 值 | 标注 |
|---|---|---|
| 理论可省调用（4–7 day 对局） | 10 – 40% | 设计预估 |
| 实测触发率（v4-flash 短局） | 偏低，待更强模型 / 更长局验证 | `实测`（诚实） |
| 对决策结果的影响 | **0**（仅在无选择余地时跳过） | `实测` |

> **横向小结**：真正经过实测验证有效的，是**检索约束、打分判别力、信息隔离、严格对局稳定性**这几步；而"分层认知 / 知识回流对净胜率的增益"仍需扩样本 A/B 验证（已在上表如实标注）。这正是后续补充实验要填的坑。
