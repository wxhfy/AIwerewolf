# AI 协作规范

> 本文件是 **AI 助手 + 人类** 协作的纪律手册。三人组每人都可能让 AI 改任何模块,因此 AI 写出来的代码必须遵守同样的规范,且**人类要负责把关**。
> 任何由 AI 直接产生的代码,**人类必须通读过才能合 PR**。

---

## 一、心智模型(读完这段才动)

**AI 是协作者,不是模块所有者**。

| 想法 | 现实 |
|------|------|
| "AI 写的代码风格肯定对" | AI 不一定读了 `skills/`,会按它训练时见过的风格写 |
| "我让 AI 跑通了测试,可以合 PR 了" | 测试通过 ≠ 设计合理,跨模块影响不在测试覆盖范围 |
| "我用同一个 prompt 多次让 AI 改" | 每次 AI 看到的上下文可能不同,结果不一致 |
| "AI 说没问题就没问题" | AI 会自信地胡说(LLM 幻觉),人类必须验证 |
| "AI 改了我没看的文件" | **必须**逐文件检查,默认 AI 改了你不知道的东西 |

## 二、开工前(给 AI 喂上下文)

让 AI 干活之前,**必须**把以下文件喂进 AI 的上下文:

### 必喂

| 文件 | 用途 |
|------|------|
| `CLAUDE.md` 或 `AGENTS.md` | 项目根级指引 |
| `SKILLS.md` | 项目业务背景 |
| `skills/README.md` | 规范索引 |
| `skills/<对应主题>.md` | 你正在改的模块对应规范 |

### 跨模块改动还要喂

| 改动类型 | 额外喂的文件 |
|----------|--------------|
| 改 Engine 模型 | `skills/40-agent-development.md`(影响 Agent)+ `skills/50-api-contract.md`(影响前端) |
| 改 API 路由 | `skills/50-api-contract.md` + `skills/30-frontend-conventions.md` |
| 改 Agent Prompt | `skills/40-agent-development.md` + 现有 prompts.py 内容 |
| 改 DB schema | `skills/20-backend-conventions.md` 第八节 + 现有 `backend/db/` |

### 喂上下文的话术示例

```
我要在 backend/agents/ 新增一个 RandomAgent。请先读以下文件,再开始:
1. /home/fyh0106/AIwerewolf/skills/40-agent-development.md
2. /home/fyh0106/AIwerewolf/backend/agents/base.py
3. /home/fyh0106/AIwerewolf/backend/agents/heuristic.py(参考实现)
4. /home/fyh0106/AIwerewolf/backend/agents/factory.py(注册点)

读完后,先列出你打算改的文件清单,我确认后再动手。
```

**不要这样说**:`"加个 RandomAgent"`——AI 会按它记忆里的风格猜,大概率不符合本项目规范。

## 三、AI 改代码前的"占位"动作

要求 AI 在动手前:

1. **列出打算读的文件**(你 review 一遍,补漏)
2. **列出打算改的文件 + 大致改法**(你 review,确认范围)
3. **指出对其他模块的影响**(AI 容易漏)

只有这三步过了再让它动手 Edit / Write。

### 错误示例

```
> 用户:加个开始游戏前的预热环节
> AI:好的,我改一下 backend/engine/game.py 加个 warmup_phase  ← 直接动手了,没列计划
```

### 正确示例

```
> 用户:加个开始游戏前的预热环节(先读规范)
> AI:我先读 skills/20、skills/40、engine/game.py、engine/phases.py、engine/models.py 五个文件
> 用户:好,读完告诉我你打算改什么
> AI:我打算在 Phase 枚举加 SETUP_WARMUP,在 phases.py 加处理函数,在 visibility.py 加对应的 phase 字符串。影响:前端 i18n 字典要加 zh/en 翻译,skills/50-api-contract.md 要更新 Phase 列表。
> 用户:批准,但 Phase 改动是重灾区,我要确认 → 让 AI 动手
```

## 四、AI 改代码时的红线

### 绝对禁止

- [ ] **不通知**人类就改 `engine/models.py`、`engine/phases.py`、`protocols/schemas.py`、`db/models.py` 这种中枢文件
- [ ] **不更新文档**就改 API / WebSocket / Schema(必须同 PR 改 `skills/50-api-contract.md`)
- [ ] **不通知**就 `git add -A` 把陌生文件加进来
- [ ] **不通知**就引入新依赖到 `requirements.txt`
- [ ] **不通知**就改 `.env`、`.gitignore`、`.claude/settings.local.json`
- [ ] 在 commit message 或代码注释里写 "Generated by Claude" / "AI-generated" / "由 Claude 自动生成" 这种**没用还露馅**的话——披露写在 **PR 描述**,代码本身不需要标注
- [ ] 把 prompts.py 全文 / 真实 API Key / `.env` 内容粘进 commit message / issue / PR 描述
- [ ] 修改 commit message 改写历史(`git rebase -i` 之类)未经人类批准

### AI 可以独立做的

- 跑 `git status` / `git diff` / `git log`(只读)
- 在 feature 分支上 commit / push
- 读项目里**任何**文件(无访问限制)
- 跑 `pytest tests/`、`python -m backend.run_demo`
- 写 / 改 / 删 feature 分支下的代码(由人类后续 review)
- 跑 grep / find 搜索

### AI 必须先问人

> **单人模式覆盖**：第一条 `git push origin main` 在单人阶段豁免（见 `CLAUDE.md` 顶部"项目运行模式"）；其他条款仍生效。

- ~~`git push origin main` 或类似改主干~~ <sup>单人模式豁免</sup>
- 创建 PR(可以草拟描述,但点"Create" 由人按)
- 改 `requirements.txt`、`.gitignore`、`pyproject.toml`(若存在)
- 删除现有测试 / 让现有测试 disable
- 改 `skills/*.md`、`CLAUDE.md`、`AGENTS.md`、`SKILLS.md`、`README.md`

## 五、AI 改完之后的审查清单

PR 包含 AI 生成代码的,人类作为**第一审稿人**必须:

- [ ] **逐文件**通读 AI 改的所有文件(不是 diff 摘要)
- [ ] 跑过 `pytest tests/`,全绿
- [ ] 改了前端的话浏览器实际点过
- [ ] 改了 LLM 的话至少跑过 1 次实际对局
- [ ] 没有引入未声明的依赖
- [ ] 没有泄漏 API Key / Prompt / 私有数据
- [ ] 没有"半实现"——所有函数都跑得通,不是 `raise NotImplementedError` 留坑
- [ ] 注释 / docstring 符合 `20-backend-conventions.md` 第十一节
- [ ] **PR 描述里披露**:哪些文件由 AI 生成、人工 review 重点

## 六、AI 的常见坑(必须警惕)

### 1. 凭印象写代码

AI 可能记得**它训练时见过的**FastAPI 风格,而不是本项目的。
**对策**:每次都让 AI 先读现有相邻文件,看代码风格再动手。

### 2. 自信地虚构 API

AI 会调用根本不存在的函数 / 字段。
**对策**:跑 `pytest` + 浏览器手测,不只看 IDE 红线。

### 3. 过度抽象

AI 喜欢加 base class、工厂、策略模式...本项目体量根本不需要。
**对策**:看到 AI 引入抽象层 / 装饰器 / metaclass,**质疑它**——"为什么不直接写?"

### 4. 添加 "fallback" / "保护性" 代码

AI 喜欢加 `try/except` 包一切,把异常吞掉。
**对策**:除非**业务上**就该降级(如 LLM 调用),否则不要让 AI 加 catch-all。
违反 `20-backend-conventions.md` "不要捕获不会发生的异常"原则。

### 5. 复读上下文

AI 写 prompt 时容易把整个 system prompt 复读进 USER 消息。
**对策**:检查 Prompt 模板,确保**仅**变化部分进 USER。

### 6. 多 Agent 改同一文件

不同 Agent / 不同 session 改同一文件,容易产生不一致风格。
**对策**:**先 git pull** 再让 AI 动手,AI 不要相信自己缓存里的旧版本。

### 7. 32MB 上下文炸裂(Claude Code 专属)

同一张图 / 大文件多次粘贴,session 历史会爆 → API 报错。
**对策**:见 `~/.claude/CLAUDE.md` "Claude Code 32MB 错误预防"段。

## 七、API Key 与秘密管理

**铁律**:

1. **API Key 只在 `.env`**,从不进代码 / 注释 / commit message / PR / Issue
2. **`.env` 必须在 `.gitignore`**(已配置,验一下)
3. **AI 不能读 `.env`**(权限上做不到拦截,但 AI 必须自律不打印内容)
4. **PR 创建前** `git log -p` 搜一遍 `sk-`、`Bearer`、`API_KEY` 关键词
5. **误提交**了 → 立刻轮换 Key → `git filter-repo` 清历史

### AI 收到 API Key 的应对

如果用户**主动**把 Key 贴进对话:

- AI **不要**重复 Key 内容("好的,我看到你的 Key 是 sk-xxx" ← 错!)
- AI **不要**把 Key 写进任何文件
- AI 应**提醒用户**这种操作有泄漏风险

## 八、AI 与跨工具协作

### Claude Code 与 Codex 同时存在

本项目同时支持 Claude 和 Codex(分别对应 CLAUDE.md / AGENTS.md):

- **不要假设另一个 AI 改了什么**——以 git 当前状态为准
- 切工具前 `git status` 看清楚,不要带着脏工作树切
- 不同 AI 对同一规范的理解可能略有差异,以 `skills/` 文件为准

### Subagent / 并行 AI

如果用 Claude 的 Task 工具或 Codex 的 multi-agent 并行:

- **每个 subagent 都要被喂同样的规范**——不能假设 "parent 读了 child 就读了"
- 多 agent 改同一文件容易冲突,**避免并行改同一目录**
- 并行结果**必须**人类汇总 review

## 九、PR 描述模板(AI 协作披露段)

```markdown
## AI 协作披露

- 本 PR 由 AI 参与生成:[ ] 是 / [ ] 否
- 使用工具:[ ] Claude Code / [ ] Codex / [ ] 其他: ___
- AI 改动范围:
  - `backend/xxx.py`(全部由 AI 生成)
  - `frontend/app.js`(AI 改了 startGame 函数)
- 人工 review 重点:
  - 验证了 Agent fallback 逻辑
  - 跑过 pytest 全绿
  - 浏览器手测 AI vs AI 一局
- 喂给 AI 的规范文件:
  - skills/20-backend-conventions.md
  - skills/40-agent-development.md
```

## 十、紧急情况:AI 把代码搞坏了

| 场景 | 应对 |
|------|------|
| AI 误删文件 / 大段代码 | `git checkout -- <file>` 或 `git reflog` 找回 |
| AI 改了 main(不应该但万一) | `git revert` 立刻回滚,**不要** `reset --hard` |
| AI 提交了秘密 | 见第七节"误提交" |
| AI 在死循环跑 LLM 烧钱 | 立刻 Ctrl+C,检查 .env 用量,必要时轮换 Key |
| AI 跑通了测试但代码很怪 | 优先信你的眼睛,质疑 AI,逐行 review |

## 十一、关于"AI 是协作者"的最后一句

AI 帮你写代码,但**你**是 PR 作者,**你**为这段代码负全责。
出了问题,不能甩锅"AI 写的"——你 push 你负责。
所以审稿要慢、要细,不要看完 diff 就 LGTM。
