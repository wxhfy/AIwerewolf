> **可自主完成一局狼人杀的 Agent Team，并进一步具备可评测、可复盘、可量化、可轻量自进化能力的 AI 狼人杀多智能体系统。**

# 1. 项目最终目标

请首先明确当前项目最终开发目标。

## 1.1 总目标

做一个能自主完成一局狼人杀的 Agent Team。

为什么选择狼人杀：

- 狼人杀具备多角色；
- 存在强信息不对称；
- 同时需要阵营协作与阵营对抗；
- 既有规则推理，也有社交博弈；
- 非常适合检验 Agent Team 的综合能力。

请判断当前仓库是否支撑以下目标：

1. **角色 Agent**  
   支持狼人、预言家、女巫、猎人、村民等常见角色，每个角色具备对应行动、策略和身份差异。

2. **对局引擎**  
   支持回合流转、夜间行动、白天发言、投票、死亡结算和胜负判定。

3. **信息隔离**  
   每个 Agent 只能看到自己身份允许的信息，不能偷看其他玩家真实身份、狼队信息、预言家查验结果或赛后真相。

4. **可观测性**  
   对局日志可查看，关键决策可追溯，能够复盘每个 Agent 在每回合的输入、输出和决策原因。

5. **加分能力**  
   提供前端 UI，支持观战或人机混战；进一步支持评测复盘、自进化或通用 Agent 能力。

## 1.2 请输出一句话项目主线

请用一句话概括当前项目的主线。推荐判断方向：

> 我们做了一个 AI 狼人杀多智能体系统，不只会玩，还能复盘每个关键决策，通过反事实评估定位 bad case，并把经过可信度、权限和适用条件过滤的策略知识回流到下一代 Agent。

请结合当前仓库实际情况判断这句话是否成立。如果不完全成立，请指出缺口。

# 5. 解释设计方案为什么这么设计

不要只罗列模块。请解释这些设计背后的原因。

至少说明以下设计：

## 5.1 为什么需要 PlayerView / Visibility 信息隔离？

请解释狼人杀的信息不对称为什么是核心挑战，以及为什么不能直接把完整 GameState 传给 Agent。

必须回答：

- 如何防止狼人看到预言家查验？
- 如何防止村民看到狼队名单？
- 如何防止 Memory / Retrieval / Prompt 泄露 hidden truth？
- 为什么需要 final_agent_input 级别测试？

## 5.2 为什么需要 BeliefTracker？

请解释 Agent 不能知道真相，只能维护“自己相信的身份概率”。

必须区分：

```text
真实身份 Truth
Agent 信念 Belief
公开声称 Claim
历史行为 Evidence
```

## 5.3 为什么需要 RoleStrategyCard？

请解释人格风格和策略决策的区别。

```text
Personality Layer：控制怎么说
Strategy Layer：控制做什么
```

说明为什么不同角色不能只共用一个 Prompt。

## 5.4 为什么需要 DecisionTrace？

请解释为什么不能只保存原始 LLM 输出，而要记录：

- visible_facts；
- candidate_actions；
- chosen_action；
- confidence；
- rationale；
- retrieved_strategy_ids；
- prompt_hash；
- model / token / latency / cost。

## 5.5 为什么需要 CounterfactualAnalyzer？

请解释为什么“只看胜负”不足以评价 Agent 决策。

示例：

- 好人赢了，但女巫可能仍然做错过；
- 狼人输了，但某只狼的倒钩策略可能是正确的；
- 某个投票动作可能是关键转折点。

## 5.6 为什么需要 L0-L4 知识可信度分层？

请解释为什么复盘不一定正确，错误复盘直接入库会污染下一局 Agent。

必须说明：

```text
不是所有复盘都可信；
不是所有可信知识都可见；
不是所有可见知识都适用于当前局面；
不是所有知识都应该永久保留。
```

## 5.7 为什么主线应该是“评测 + 复盘”？

请判断当前项目更适合主打哪个进阶方向：

- A. 通用 Agent；
- B. 评测 + 复盘；
- C. 自进化 Agent。

请说明理由，并指出如果主打错误方向会有什么风险。

# 7. 进阶课题要求与验收方式

进阶课题三选一。请检查当前项目最适合选择哪个方向，并说明原因。

## 7.1 A. 通用 Agent

### 核心问题

Agent 能否根据目标自主生成 / 修改代码、测试、部署？

### 具体要求

- 能读懂自身代码；
- 能改代码并提交；
- 能自动构建测试重启；
- 失败可回滚；
- 过程可审核。

### 验证方式

以“玩一局狼人杀”为目标，系统自主演化出可运行的 Agent Team。

### 请判断

当前项目是否真的具备：

```text
读代码 → 改代码 → 自动构建测试 → 失败回滚 → 审计记录
```

如果没有，请不要把它作为主赛道。

## 7.2 B. 评测 + 复盘

### 核心问题

如何系统化评价 Agent 决策质量，而不只看胜负？

### 具体要求

- 多维评测：发言 / 投票 / 技能；
- 关键决策复盘；
- 反事实推演；
- 结构化报告；
- Leaderboard。

### 验证方式

构造含明显失误的对局，系统能定位失误并给出改进建议。

### 请检查

- 是否能生成 ReviewReport；
- 是否有 PerStepScorer；
- 是否有 CounterfactualAnalyzer；
- 是否能定位 bad case；
- 是否有 Leaderboard；
- 是否能区分不同模型 / Agent 版本能力。

## 7.3 C. 自进化 Agent

### 核心问题

Agent 能否在多局迭代中持续提升胜率？

### 具体要求

- 自动“对局 → 分析 → 调整 → 再对局”闭环；
- 策略适应性；
- A/B 对战验证；
- 版本回溯。

### 验证方式

终局 Agent 与初始 Agent 对战 20 局，胜率显著提升。

### 请检查

- 是否有 StrategyKnowledgeDoc；
- 是否有知识回流；
- 是否有知识可信度分层；
- 是否有 A/B Tournament；
- 是否有版本记录；
- 是否真的有 20 局对战提升数据。
