## /goal

请你全面阅读当前 AI Werewolf / AI 狼人杀项目代码与文档，基于真实仓库内容完成一次：

```text
系统目标确认 → 架构理解 → 测试量化 → 满分导向验收指标设计 → 评分标准对齐 → 小步可执行改进 → 大改方案沉淀
```

你的任务不是只判断“能不能跑”，也不是只判断“是否达到 MVP”。你必须以比赛评分细则中的**满分档**作为验收目标，检查当前项目距离满分还有多远，并把能小改的部分直接改掉，把大改部分写入 backlog。

本项目的核心不是做一个普通狼人杀游戏，而是做一个：

> **可自主完成一局狼人杀的 Agent Team，并进一步具备可评测、可复盘、可量化、可轻量自进化能力的 AI 狼人杀多智能体系统。**

> 我们做了一个 AI 狼人杀多智能体系统，不只会玩，还能复盘每个关键决策，通过反事实评估定位 bad case，并把经过可信度、权限和适用条件过滤的策略知识回流到下一代 Agent。

# 1. 满分导向原则

从现在开始，所有验收指标必须分成三层：

| 层级 | 含义 | 用途 |
|---|---|---|
| MVP 线 | 项目能基本跑通，满足最低提交要求 | 判断是否能交付 |
| 及格线 | 达到评分细则中的及格档 | 判断是否不被明显扣分 |
| 满分线 | 对齐评分细则中的满分档 | 判断是否具备冲奖竞争力 |

最终报告和验收表必须以**满分线**作为目标。MVP 和及格线只作为中间状态说明。

请不要只写：

```text
能跑通一局。
```

要写成：

```text
MVP：能跑通一局标准狼人杀。
及格：核心流程稳定，部分边界可接受，基本信息隔离。
满分：批量对局稳定，边界处理完善，信息隔离经测试无泄露，日志可复现，前端和报告能让非技术评委看懂。
```

# 2. 改动模式：审查 + 小改 + 大改建档

你需要将发现的问题分成四类处理。

| 等级 | 类型 | 允许动作 | 示例 |
|---|---|---|---|
| S0 | 只记录，不修改 | 写入报告即可 | 长期优化想法、主观风格建议 |
| S1 | 安全文档小改 | 可以直接修改并提交 | 文档补充、验收表补全、报告模板补齐 |
| S2 | 小范围代码修复 | 可以直接修改，但必须最小 patch、可测试、可回滚 | 明显 typo、错误 fallback 顺序、日志字段补充、测试断言补充 |


每个问题必须输出：

# 3. 项目最终目标与基础要求

## 3.1 总目标

做一个能自主完成一局狼人杀的 Agent Team。

为什么选择狼人杀：

- 多角色；
- 信息不对称；
- 阵营协作；
- 阵营博弈；
- 规则推理；
- 社交欺骗；
- 很适合检验 Agent Team 的综合能力。

## 3.2 基础课题要求

| 模块 | 要求 | 验证方式 |
|---|---|---|
| 角色 Agent | 支持狼人、预言家、女巫、猎人、村民，各自有独立决策逻辑 | 各角色行为符合身份设定，策略有差异 |
| 对局引擎 | 回合流转、夜间行动、发言、投票、胜负判定 | 能完整跑完一局标准狼人杀 |
| 信息隔离 | 每个 Agent 只能看到自己身份允许的信息 | 验证狼人看不到查验结果，村民看不到夜间行动和狼队名单 |
| 可观测性 | 对局日志可查看，关键决策可追溯 | 每回合输入、输出、决策均有记录 |
| 前端 UI | 支持观战或人机混战 | 有可操作的前端界面，至少支持观战模式 |

请逐项输出：

- MVP 线；

- 满分线；
- 当前状态；
- 证据路径；
- 可小改项；
- 大改待办；
- 验收方式。

## 5.1 单 Agent 能力，20%

### 评分考察点

- Prompt 设计质量；
- 角色人设；
- 思维链或结构化决策链；
- few-shot 或策略示例；
- 迭代调优过程；
- 各角色策略差异度；
- 决策推理链路可追溯性；
- 量化评估手段；
- bad case 识别与分析。

### 满分档原意

Prompt 精细且有迭代调优痕迹，不同角色行为差异显著；决策有可追溯推理链路，有量化评估手段，能识别并分析 bad case。

### 满分验收标准

| 子项 | 满分验收标准 | 量化方式 |
|---|---|---|
| 角色 Prompt / 策略独立性 | 狼人、预言家、女巫、猎人、村民均有独立 Prompt 或 StrategyCard | 检查文件路径，5/5 角色覆盖 |
| 角色行为差异 | 不同角色在发言、投票、技能、信息披露上有显著差异 | 行动分布、策略模板、样例对局对比 |
| 决策可追溯 | ≥95% Agent 决策有结构化 DecisionTrace | 统计 AgentDecision / DecisionTrace 覆盖率 |
| 候选动作记录 | 决策记录包含 candidate_actions 和选择理由 | 抽样检查日志或数据库 |
| 信念变化记录 | 决策前后有 belief_delta 或等价记录 | 抽样检查 BeliefTracker 输出 |
| Prompt 迭代痕迹 | 有 prompt_version、agent_version、bad case 修复记录 | 检查版本字段、变更日志、文档 |
| bad case 能力 | 能识别 Golden Cases 中明显错误，TPR ≥80%，并报告 CI | 运行 Golden Cases 校准 |
| 量化评估 | 至少有发言、投票、技能、决策质量指标 | 检查 eval 报告 |

## 5.2 多 Agent 协作与系统设计，20%

### 评分考察点

- 上下文管理；
- 每个 Agent 的对话历史；
- 公共 / 私有信息维护与传递；
- 技能管理；
- 查验、毒药、开枪等技能抽象与调度；
- Agent 间博弈行为：伪装、欺骗检测、归票、站队；
- 多 Agent 通信与协调机制。

### 满分档原意

上下文管理清晰，公共 / 私有信息分离、历史管理合理；技能调度抽象良好，加新角色 / 技能改动小；Agent 间有明确博弈行为，例如归票、交叉验证、选择性暴露等。

### 满分验收标准

| 子项 | 满分验收标准 | 量化方式 |
|---|---|---|
| 公共 / 私有信息分离 | GameState、PlayerView、TeamView、Post-game Truth 明确隔离 | 代码路径 + 信息隔离测试 |
| final_agent_input 安全 | PlayerView + Memory + Retrieval + Prompt 全链路无 hidden truth | 泄露测试为 0 |
| 历史管理 | 每个 Agent 有可控历史、摘要或 Memory，不粗暴无限拼接 | 检查 Memory / context 构建 |
| 技能抽象 | 技能有统一接口或清晰分发逻辑，新增角色改动小 | 检查 Skill / ActionValidator / RoleRegistry |
| 狼队协作 | 狼队有合法 WolfTeamView、刀人协商、战术分配或统一口径 | 检查代码和日志 |
| 好人协作 | 有归票、站队、交叉验证、验人链或可信度传播 | 检查 BeliefTracker / 投票逻辑 |
| 欺骗与反欺骗 | 狼人能伪装，好人能检测矛盾 | 样例对局 + bad case |
| 多 Agent 通信 | 公共发言、私有队伍信息、投票历史都能进入合法上下文 | final_agent_input 抽样 |
