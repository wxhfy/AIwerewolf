AI Werewolf 人机对战模式设计文档

本文档用于指导人机对战页面设计与实现：复用 AI 观战页成熟的视觉、组件、转场和状态同步机制，但将页面核心从“事件时间线”切换为“当前行动舞台”。

模式：人机对战 核心：当前行动者舞台化 原则：复用视觉，不复用观战逻辑 协议：沿用 AI 对战数据格式 状态：统一派生，严格隔离

1. 设计目标

人机对战模式是“玩家亲自参与，其余角色由 AI 扮演”的狼人杀模式。

它不是 AI 观战页的简单加按钮版本，也不是重新做一套完全不同的页面。正确方向是：

复用 AI 观战页成熟的视觉、组件、转场和状态同步机制，但把中间核心从“事件时间线”改成“当前行动舞台”。

AI 观战页的核心是“看一局 AI 自动对战”，所以它适合以 EventTimeline、打字机、事件顺序播放作为主体验。人机对战页的核心是“我正在参与游戏”，所以它必须优先回答：

  • 现在是第几天或第几夜；
  • 当前是什么阶段；
  • 现在轮到谁；
  • 我是否需要操作；
  • 如果需要操作，我应该点什么；
  • 如果不需要操作，我在等待什么；
  • 哪些信息对我可见，哪些信息必须隐藏。

因此，人机对战页的核心设计原则是：当前行动者舞台化。

  • 谁在发言，谁上台；
  • 哪个角色正在行动，哪个角色成为舞台主题；
  • 玩家能操作时，底部给出明确操作；
  • 玩家不能操作时，底部给出明确等待；
  • 历史事件进入事件日志，不占据主舞台。

2.2 不能直接复用的部分

AI 观战页的 EventTimeline 不应作为人机页主舞台。

原因是：

  • AI 观战页是“事件播放”；
  • 人机页是“玩家参与”；
  • AI 观战页的 displayPhase 可以跟随最早未完成的聊天消息；
  • 人机页的阶段必须跟随现有游戏快照中的真实 phase，并由前端基于当前玩家、角色和行动者派生可操作状态；
  • 人机页不能因为打字机没播完就阻塞玩家操作，也不能因为事件流顺序导致顶部、玩家卡、操作面板不同步。

因此，人机页应将 EventTimeline 降级为事件日志 / 发言记录的内容来源，主舞台应由新的 HumanStageArea 承担。

3. 页面总体结构

人机对战页面采用固定牌桌结构：

顶部状态栏 GameHeader
左右玩家席位 PlayerRail
中间当前行动舞台 HumanStageArea
底部玩家操作面板 HumanActionPanel
事件日志抽屉 HumanEventDrawer
覆盖层 PhaseOverlayCoordinator / DramaticOverlay / GameEndPanel

页面结构示意：

┌──────────────────────────────────────────────┐
│ GameHeader：房间号 / 第几天夜 / 存活 / 身份 │
├──────────┬──────────────────────┬──────────┤
│ 左玩家栏 │      当前行动舞台     │ 右玩家栏 │
│ Player  │   HumanStageArea      │ Player  │
│ Rail    │   当前主角 / 当前任务 │ Rail    │
├──────────┴──────────────────────┴──────────┤
│ HumanActionPanel：操作区 / 等待区            │
└──────────────────────────────────────────────┘

中间主舞台只展示当前阶段最重要的信息。历史发言、阶段日志、投票记录、系统事件进入事件日志抽屉，不再堆在主舞台中。

4. 核心交互原则

4.1 阶段展示和操作权限分离

错误逻辑：我是预言家，所以夜晚只显示预言家行动，跳过守卫、狼人、女巫。
正确逻辑：所有玩家都能看到完整阶段流转。只有对应身份玩家在对应阶段可操作。无操作权限的玩家看到等待态。

当前玩家是预言家时，夜晚仍然应该看到：

第1夜 · 夜幕降临
第1夜 · 守卫行动：守卫正在行动，请等待
第1夜 · 狼人行动：狼人正在行动，请等待
第1夜 · 预言家行动：轮到你了，请选择查验对象
第1夜 · 女巫行动：女巫正在行动，请等待
第2天 · 天亮结算

canAct 只能决定是否显示操作按钮，不能决定是否显示阶段。

4.2 当前行动者上台

白天发言、遗言、竞选发言、警长发言等“人说话”的阶段，应突出当前玩家本人。夜晚行动阶段，应突出当前角色主题。

阶段类型主舞台展示重点
白天发言当前发言玩家立绘 / 大头像 / 半身图。
遗言阶段出局玩家立绘 / 大头像。
狼人行动狼人主题舞台。
预言家行动预言家主题舞台。
女巫行动女巫主题舞台。
守卫行动守卫主题舞台。
投票结果 / 天亮结算 / 游戏结束结果卡片，不抢人物舞台。

如果没有完整立绘资源，可以降级为：放大头像 + 身份徽记 + 主题背景。不能因为缺少立绘就退回日志页。

4.3 主舞台只展示当前重点

主舞台不能同时出现投票结果、遗言、夜幕降临、守卫行动、狼人行动、预言家行动、历史发言日志。这些内容必须按阶段逐个展示。

每一刻页面只允许有一个主阶段：

第3天 · 投票结果 第3天 · 遗言阶段 第3夜 · 夜幕降临 第3夜 · 守卫行动中 第3夜 · 狼人行动中 第3夜 · 预言家行动中 第3天 · 天亮结算

不能出现“第3天 + 第3夜行动”混杂。

5. 顶部状态栏设计

顶部状态栏只展示全局状态，不展示任何调试字段。

需要显示

  • Logo
  • 房间号
  • 模式：人机对战
  • 当前第几天 / 第几夜
  • 存活人数
  • 当前阶段
  • 当前玩家身份
  • 当前视角
  • 设置入口

禁止显示

  • D0 / D1 / D3
  • NIGHT_WOLF_ACTION
  • NIGHT_SEER_ACTION
  • phase xxx
  • loading failed

示例：

AI Werewolf｜房间 e6b9950a｜人机对战
第1夜｜存活 10/10｜狼人行动中
身份：预言家｜玩家视角
第3天｜存活 5/7｜遗言阶段
身份：村民｜玩家视角

建议增加统一映射函数：

getHumanPhaseDisplay(state): {
  title: string;
  subtitle: string;
  stageLabel: string;
  theme: "day" | "night" | "neutral";
  actionText?: string;
}

所有顶部文案、中间标题、事件日志阶段名都从这个映射派生，避免各组件自己拼文案。

6. 玩家席位设计

左右玩家席位继续复用 PlayerRail + PlayerCard 的视觉结构，但在人机页中，玩家卡是“局势状态容器”，不是静态名单。

每张玩家卡应展示：

玩家编号 头像 昵称 简短人设 存活 / 死亡 / 出局 当前行动 正在发言 遗言中 已发言 已投票 已行动 警长 警长候选人 被当前玩家选中 是否可点击

状态标签示例：

正在发言
遗言中
已出局
已投票
当前行动
已提交
死亡
警长
候选人
  • 当前主舞台是谁，该玩家卡同步高亮；
  • 死亡/出局不能只降低透明度，必须显示明确标签；
  • 被选为目标的玩家卡和目标网格要同步选中态；
  • 普通玩家视角不能通过玩家卡看到其他玩家真实身份；
  • 主持视角可以看到全身份，但必须有明显提示。

7. 主舞台阶段模板

7.1 准备阶段

主舞台展示：

玩家正在入场
AI 玩家正在准备
10/10 已就绪
等待游戏开始

底部操作：未准备显示“准备”；已准备显示“已准备，等待开始 / 取消准备”；房主或可启动者显示“开始游戏”。

7.2 身份揭示阶段

使用居中身份卡。身份卡内容示例：

你的身份：预言家
能力：每晚可以查验一名玩家的阵营。
提醒：合理隐藏身份，关键时刻带队。
按钮：继续
你的身份：狼人
能力：夜晚可与狼队友商议并击杀一名玩家。
狼队友：3号、7号
提醒：白天隐藏身份，谨慎发言。
按钮：继续

身份揭示必须遵守信息隔离：玩家只看到自己的身份；狼人额外看到狼队友；公开视角隐藏身份；主持视角显示完整身份。

7.3 夜幕降临过渡

“天黑请闭眼”只是夜晚开始的短暂过渡，不是夜晚所有阶段的常驻背景文案。

第1夜 · 夜幕降临
天黑请闭眼

过渡期间不显示具体角色操作、不显示目标选择、不显示查验、击杀、守护、用药面板。过渡结束后，再进入具体行动阶段。

7.4 守卫行动

守卫本人看到：

第1夜 · 守卫行动
请选择今晚守护对象

底部：
目标选择网格
确认守护

其他玩家看到：

第1夜 · 守卫行动
守卫正在行动，请等待

不能显示守护目标。

规则限制：守卫不能连续两晚守护同一名玩家。前端应在目标选择网格中禁用上一晚已守护的目标（灰色 + "已连续守护"提示），防止玩家误选后提交到后端才报错。
7.5 狼人行动

狼人本人看到：

第1夜 · 狼人行动
狼人正在商量今晚的击杀目标

底部：
请选择击杀目标
目标选择网格
确认击杀

可见信息包括狼队友、狼队友选择状态、狼人内部目标。非狼人只能看到“狼人正在行动，请等待”，不能显示狼队友、击杀目标、狼人讨论内容。

7.6 预言家行动

预言家本人看到：

第1夜 · 预言家行动
请选择一名玩家查验身份

底部：
目标选择网格
确认查验

查验完成后，仅预言家可见：

查验结果：5号 夏知白 是好人阵营
查验结果：5号 夏知白 是狼人阵营

其他玩家只看到“预言家正在行动，请等待”，不能显示查验目标和查验结果。

7.7 女巫行动

女巫本人看到：

第1夜 · 女巫行动
昨夜有人被袭击
是否使用解药？
是否使用毒药？

底部：
使用解药 / 不使用解药
使用毒药 / 不使用毒药
选择毒杀目标
确认

需要记录解药是否可用、毒药是否可用、是否已用药、毒杀目标、是否已提交。其他玩家不能看到死亡目标、用药选择、毒药目标，除非规则允许公开。

7.8 天亮结算

夜晚结束后，先进入天亮结算，不要直接跳到发言日志。

第2天 · 天亮了
昨夜死亡：无人
平安夜
即将进入发言阶段
第2天 · 天亮了
昨夜死亡：4号 韩书妍
即将进入遗言阶段

死亡公告可以复用 DramaticOverlay。公告结束后，再进入遗言或发言阶段。

7.9 警长竞选阶段
第1天 · 警长竞选
是否参与警长竞选？

底部：
我要竞选
不竞选

已选择后：
已提交，等待其他玩家选择

竞选发言时，主舞台显示当前竞选发言玩家的大立绘，底部根据是否轮到自己显示发言输入或等待状态。

7.10 白天发言阶段

白天发言是人机页最重要的舞台体验。当轮到某玩家发言，主舞台显示当前发言玩家大立绘 / 大头像 / 半身形象、玩家编号、昵称、正在发言状态、当前发言内容和最近上下文气泡。

如果发言者是 AI，发言内容可复用 useTypewriter 做打字机效果，但打字机只控制当前发言文本，不控制整页 phase。

玩家本人发言：
发言输入框
提交发言

其他玩家发言：
等待 5号 赵峰 发言中
7.11 遗言阶段

遗言阶段必须独立于普通发言阶段。

顶部：
第3天｜遗言阶段

主舞台：
2号 卡卡 正在遗言

当前玩家是遗言玩家：
遗言输入框
结束遗言

其他玩家：
等待 2号 卡卡 遗言结束

禁止出现“顶部：1号正在发言，主舞台：2号正在遗言”。

7.12 投票阶段
第3天 · 投票放逐
请选择你要投票放逐的玩家
已投票 3/5

底部：
目标选择网格
确认投票
  • 未选择目标时，确认按钮置灰；
  • 选择后显示”已选择：X号 玩家名”；
  • 提交后显示”已投票，等待其他玩家”；
  • 死亡或出局玩家不可投票，只能观战。
警长票权：警长的投票计为 1.5 票。前端在投票结果和票数统计中应体现警长票权。警长投票的目标玩家在票数计算时 +1.5 而非 +1。后端 VOTE_CAST 事件已包含 vote_weight 字段，前端直接使用即可。
7.13 投票结果阶段

投票结束后必须使用结果卡，不要把结果写成几行日志。

投票结果
2号 卡卡 被放逐
票数：3票
即将进入遗言阶段
投票结果
无人被放逐
即将进入夜晚

结果卡结束后，再进入遗言或夜幕降临。

7.14 游戏结束阶段

复用 GameEndPanel，人机页额外展示：

游戏结束
好人阵营胜利
你的身份：预言家
你所在阵营获胜
查看复盘
再来一局
返回大厅

8. 底部操作面板设计

底部操作面板是人机页的核心交互区，不是日志尾部。

它根据以下状态渲染：

phase
currentPlayerRole
currentActor / currentSpeaker
canAct
actionView（前端派生）
submittedStatus
selectedTarget

8.1 可操作状态

按钮必须是具体业务动作，不能只写“提交”。

确认击杀 确认守护 确认查验 提交发言 结束遗言 确认投票 使用解药 使用毒药 跳过

8.2 等待状态

不是自己行动时，底部显示明确等待：

等待守卫行动中
等待狼人行动中
等待预言家行动中
等待女巫行动中
等待 5号 赵峰 发言中
等待 2号 卡卡 遗言结束
等待其他玩家投票

8.3 已提交状态

提交后不能继续编辑。

已提交，等待阶段推进
已选择：5号 夏知白
等待其他玩家

8.4 目标选择状态

所有目标选择类操作统一使用目标网格：狼人击杀、守卫守护、预言家查验、女巫毒杀、投票放逐、猎人开枪。

目标卡需要区分可选、不可选、已选、死亡、已出局、自己、当前高亮。未选择时按钮置灰，选择后按钮可点击。

9. 事件日志设计

事件日志作为抽屉或侧边面板存在，不占据主舞台。

记录内容包括：

玩家加入 玩家准备 身份分配 夜幕降临 守卫行动完成 狼人行动完成 预言家行动完成 女巫行动完成 天亮结果 发言顺序 警长竞选 投票进度 投票结果 放逐结果 遗言开始 遗言结束 游戏结束

事件日志必须按视角过滤：玩家视角不能看到隐藏身份、夜间目标、查验结果、用药选择；公开视角隐藏所有真实身份；主持视角可以看到完整日志；AI 调试信息不能显示在普通玩家日志里。

10. 数据流设计

重要约束：人机对战不重新设计前后端交互协议，不要求后端返回新的数据结构。它应沿用 AI 对战模式已有的房间状态、WebSocket 快照流、players、phase、events、logs 等数据格式。人机模式本质上只是把一个 AI 玩家替换为真人玩家。

人机对战的数据流应理解为对现有 AI 对战数据流的前端参与式适配：

用户在大厅选择人机对战
→ 创建房间
→ 准备阶段生成 AI 玩家与真人玩家所在席位
→ 进入 app/room/[id]/human/page.tsx
→ 沿用现有 WebSocket /ws/rooms/{id} 快照流
→ 后端继续推送与 AI 对战一致的 gameState / phase / players / events / logs
→ 前端基于现有 phase、玩家身份、当前行动者、事件和提交结果派生“当前玩家是否需要操作”
→ 需要真人输入时，前端展示 HumanActionPanel
→ 玩家通过现有 action 提交流程提交发言、投票或技能选择
→ 游戏继续按原有状态流推进
→ 游戏结束后复用 GameEndPanel
模式数据流特征
AI 观战模式后端持续推进，前端按事件流进行观战叙事。
人机对战模式沿用同一套数据快照；前端把真人玩家所在席位渲染为可参与、可操作的舞台页面。

因此，设计重点不是新增后端协议，而是前端基于现有数据结构做派生渲染：阶段舞台、操作面板、玩家卡状态、事件日志抽屉和视角展示。

11. 前端状态模型

建议构建统一 HumanGameViewState，所有 UI 从它派生。

12. 信息隔离设计

12.1 玩家视角

玩家不能看到其他玩家真实身份、其他阵营夜间目标、其他玩家查验结果、女巫用药选择、守卫守护目标、AI 内部推理和主持视角数据。

12.3 主持视角

主持视角可以看到完整身份和完整行动信息，但必须显示：

主持视角｜身份信息仅主持人可见

17. 数据协议约束

本节为硬性约束：人机对战页面不干涉后端返回数据格式，不重新设计前后端交互协议，不要求后端新增 public_state、private_state、host_state、pending_action 等新结构。

人机对战应沿用 AI 对战模式已有的数据交换格式。它的本质不是一套新的协议，而是在原有 AI 对战流程中，将一个 AI 玩家替换为真人玩家。

17.1 正确理解

  • 沿用现有房间状态和 WebSocket 快照流；
  • 沿用现有 gameState、phase、players、events、logs 等数据结构；
  • 真人玩家仍然作为一个普通 player 参与原有游戏流程；
  • 真人玩家的发言、投票、夜间行动应通过现有 action 提交流程进入游戏状态；
  • 前端基于现有数据派生“是否轮到我”“我能做什么”“应该展示哪个舞台”。

17.2 前端应该做的事情

前端需要在不改变数据格式的前提下，完成以下派生和渲染适配：

从现有 phase 派生当前阶段文案 从现有 players 派生玩家卡状态 从当前玩家 id / role 派生操作权限 从当前行动者 / 发言者派生主舞台主角 从 events / logs 派生事件日志 从提交结果派生已提交 / 失败 / 等待状态 在 UI 层进行视角展示适配

17.3 禁止方向

  • 不要要求后端为人机模式单独返回一套新结构；
  • 不要把页面设计建立在新增 pending_action 协议的前提上；
  • 不要要求后端新增 public_state / private_state / host_state；
  • 不要让前端页面改造反向牵引后端协议重构；
  • 不要为了真人玩家操作破坏 AI 对战模式已有的数据流。
最终原则：后端数据格式保持 AI 对战模式一致；前端基于现有数据结构做参与式页面适配。

17.4 状态同步策略

关键决策：人机模式不使用 WebSocket 流，完全由 POST 请求/响应驱动。

理由：

  • POST /start 和 POST /action 已经返回完整的 GameState 快照；
  • 如果同时启用 WebSocket 推送，会与 POST 响应产生竞态——WebSocket 可能推送旧快照覆盖 POST 返回的最新状态，导致 pending_input 丢失或 phase 回退；
  • 人机模式是回合制（人等输入→提交→后端继续），不需要实时流。

实现要点：

  • 人机页面不初始化 useRoomStream 的 WebSocket 连接；
  • 所有状态更新仅来自 handleHumanAction 的 POST 响应和 startHumanGame 的 POST 响应；
  • setGameState(snapshot) 是唯一的状态更新入口；
  • 如果未来需要混合模式，必须给每个快照带时间戳或版本号，前端拒绝 timestamp 更旧的快照。
// 竞态保护示例（如未来启用 WebSocket）
          function setGameStateSafe(snapshot) {
            if (snapshot._ts <= lastSnapshotTs) return; // 拒绝旧数据
            lastSnapshotTs = snapshot._ts;
            setGameState(snapshot);
          }

17.5 事件可见性过滤

现有代码缺陷：CompactEvents / EventTimeline 完全不检查 visibility 和 visible_to 字段，玩家可以通过事件日志看到夜间行动目标。这是身份泄露 bug。

修复方案：

  • 在事件渲染前，根据当前 viewMode、currentPlayerId 过滤事件；
  • visibility === "private" 的事件，只有 visible_to 包含当前 playerId 或 viewMode 为主持视角时才渲染；
  • visibility === "public" 的事件，所有视角可见；
  • 公开视角下，NIGHT_ACTION 类型的事件隐藏目标信息。
function filterEvents(events, viewMode, playerId) {
            return events.filter(e => {
              if (viewMode === "moderator") return true;           // 主持全可见
              if (e.visibility === "public") return true;          // 公开事件
              if (e.visibility === "private") {
                return e.visible_to?.includes(playerId);           // 仅可见列表中包含当前玩家
              }
              return false;
            });
          }

此过滤必须同时应用于主舞台的事件引用和事件日志抽屉。

17.6 死亡玩家状态机

玩家死亡后的三态模型：

状态触发条件页面行为
遗言中刚被放逐 && 有遗言规则显示遗言输入框；可提交遗言；顶部标记"遗言阶段"
猎人开枪死亡原因=投票 && 角色=猎人显示目标选择；可选开枪或放弃
观战遗言结束 / 无遗言 / 被狼杀主舞台显示"你已出局，正在观战"；操作面板隐藏；仍能看到公开阶段流转；不能投票或夜间操作

关键规则：

  • 死亡玩家的 canAct 始终为 false（除非正在遗言或猎人开枪）；
  • 观战玩家的信息可见范围 = 玩家视角（不因死亡而获得更多信息）；
  • 不能出现"等待其他玩家..."的永久等待态。

17.8 女巫两阶段行动处理

女巫行动涉及两个决策：解药（save）和毒药（poison + target）。后端 pending_input 一次只支持一个 action_type。

两种可行方案：

方案描述复杂度
A. 两次 pending_input后端先发 WITCH_SAVE 请求 → 玩家选择 → 后端再发 WITCH_POISON 请求。前端每次只展示一个决策。低
B. 内联两阶段前端在同一个面板内先展示解药开关，玩家确认后展示毒药目标选择。中

建议先确认后端当前行为属于哪种，再决定前端实现。如果后端已按方案 A 实现，则不需要前端支持内联两阶段。

17.9 错误恢复与刷新恢复

身份揭示恢复

问题：revealDone 是本地 useState，刷新后丢失，重新弹身份卡。

修复：角色揭示弹窗改用手动确认按钮（"继续"）替代 3 秒自动消失。玩家点击"继续"后不再弹出。如果刷新页面，身份信息仍可从 gameState.players 重新派生但弹窗不再自动触发。

ActionPanel 错误恢复

问题：submit() 立即设置 submitted=true 禁用所有输入；如果 handleHumanAction 抛异常，用户无法重试。

修复：ActionPanel 新增 error prop。当父组件传入非空 error 时，重置 submitted=false，恢复输入，显示错误提示和"重新提交"按钮。

// ActionPanel 新增逻辑
          useEffect(() => {
            if (error) {
              setSubmitted(false);
              setTimerActive(hasTimer);
            }
          }, [error]);

行动提交失败

状态行为
提交失败保留当前选择（targetId / speech 不清空）；显示"提交失败，请重试"；按钮恢复可点击
提交成功按钮变为 spinner + "已提交"；等待新 pending_input 到达自动重置
网络超时显示"请求超时，请检查网络后重试"

17.10 SKIP 语义定义

哪些阶段允许跳过：

阶段可跳过？跳过含义
警徽竞选发言是不竞选，提交空发言
自由发言是本轮不发言
遗言否必须发言（或后端允许空发言=放弃遗言）
狼人击杀否必须选择目标（或后端允许空目标=放弃击杀）
预言家查验否必须选择目标

17.11 补充阶段模板

猎人开枪

猎人本人看到：选择开枪目标 / 放弃开枪。其他玩家看到：猎人在选择是否开枪，请等待。

白狼王自爆

白狼王本人看到：是否自爆？选择带走的目标玩家。其他玩家看到：白狼王正在行动，请等待。

警徽移交

持有警徽的出局玩家看到：选择警徽继承人 / 撕毁警徽。其他玩家看到：警徽正在移交，请等待。

警徽移交完成后，主舞台显示"警徽已移交给 X 号玩家"或"警徽已销毁"。

18. 必须避免的问题

  • 顶部显示第3天，中间却是第3夜；
  • 页面显示天黑请闭眼，但底部已经出现预言家查验；
  • 玩家是预言家时，只显示预言家行动，跳过其他夜晚阶段；
  • 正在发言的是 A，主舞台却展示 B 的遗言；
  • 投票结果和夜晚流程混在同一屏；
  • 事件日志占据主舞台；
  • D3、NIGHT_WOLF_ACTION 等内部状态出现在页面；
  • 未选择目标时确认按钮可点击；
  • 已提交后还能继续编辑；
  • 公开视角看到真实身份；
  • 普通玩家看到主持视角数据；
  • 加载失败和正常游戏状态同时显示；
  • 打字机播放进度导致真实 phase 展示错乱；
  • AI 发言结束前，玩家操作状态被错误阻塞；
  • 事件日志泄露夜间行动目标；
  • 为了人机页面要求后端新增一套不同于 AI 对战的数据格式；
  • 把真人玩家设计成独立协议对象，而不是原有 players 中的一个参与者。

20. 总结

人机对战页的本质是参与式狼人杀页面，不是 AI 观战日志页。

它应该继承 AI 观战页成熟的视觉体系、玩家卡、转场动画、覆盖层、错误兜底和统一状态派生思想；但页面核心必须从 EventTimeline 切换为 HumanStageArea。

同时，人机对战页不改变前后端数据交换格式。真人玩家本质上只是原有 AI 对战流程中的一个玩家席位，前端应基于现有快照数据做舞台化、操作化和视角化渲染适配。

最终体验应该是：白天谁发言，谁上台；遗言谁在说，谁上台；夜晚哪个角色行动，哪个角色成为舞台；玩家能操作时，底部给出明确操作；玩家不能操作时，底部给出明确等待；历史信息进入事件日志；所有 UI 从统一状态派生；所有隐藏信息按视角严格隔离。

如果做到这些，人机对战页才会真正像一个可玩的狼人杀牌桌，而不是 AI 观战页的临时改造版。
